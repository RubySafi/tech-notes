Understanding the Git Workflow

Git helps track changes in your code. Understanding its workflow is crucial for using Git effectively. Below is the basic workflow you'll follow when working with Git.

  • First Example: Git Environment with VSCode
  • Second Example: Git Environment with VSCode without remote repository
  • Third Example: How to Link a Local Git Repository to GitHub with VSCode

Git Terminologies and Workflow Overview

Before explaining the workflow, I recommend familiarizing yourself with some basic terminologies and their relationships. Below are brief explanations and illustrations of these concepts. While they may not be strictly accurate in every technical sense, they will help you grasp the Git workflow in the beginning.

  • Repository: A place to store source code/projects, including all approved versions. There are two types of repositories: remote and local.
  • Working Directory: A local area where you edit your source code/projects. It always contains the latest version, and no approval is needed.
  • Staging Area (Index): A temporary area where source code/projects are stored before approval.
  • Staging: The process of moving a file from the working directory to the staging area.
  • Commit: The process of approving and saving changes from the staging area to the local repository. After committing, the staging area is cleared.
  • Synchronize: The process of updating the remote repository with the latest version from the local repository.

First Example: Git Environment with VSCode

This first example of a Git workflow consists of the following steps:

  1. Create a repository on GitHub.
  2. Clone the remote repository to your working directory.
  3. Freely create/edit files inside your working directory.
  4. After making some edits, stage the changes and commit them.
  5. Synchronize your changes with the remote repository. Then return to step 3.

This approach requires only a few commands, making it beginner-friendly. It's the best starting point to explore the world of Git. Let's break down the process.

1. Create a repository on GitHub

Go to GitHub and log into your account. Once logged in, click the + sign in the top-right corner and select New repository. (You can also create a new repository from the "Repositories" tab.) Do the following setup:
  • Choose a name for your repository, such as my-first-repo.
  • Leave the description field empty for now, but it's always helpful to add one later to explain what your project is about.
  • Toggle the repository type to Private. [What is Public?]

    "Public" means anyone can view your code, so avoid including sensitive or private information. In the future, you can adjust this setting according to your needs.

  • Check the box to Initialize this repository with a README to add a basic file that explains your project.
  • Leave the .gitignore and license options set to None. [What is them?]
    • A .gitignore file specifies which files should be excluded from version control. This is useful for automatically generated files, compiled outputs, or private configuration files. You can add a .gitignore file manually at any time, but it's an advanced setting.
    • A license defines how others can use your repository. For example, the MIT License allows free use with minimal restrictions. This is an advanced topic, as understanding the details of a license requires some legal knowledge. If you're interested, take some time to research licenses before making your repository public. It's not essential for beginners, but it's good to understand the basics as you grow in your coding journey.
After configuring these settings, click Create repository. Then, you can move to the step to clone repository.

2. Clone the remote repository to your working directory

The next step is to bring the new repository to your local machine so you can work on it. This is called "cloning" the repository.

On the repository page on GitHub, click the green Code button, then copy the URL under Clone with HTTPS. (You can also clone using SSH if you have set up SSH authentication.)

Open Git Bash. Change your current directory to the place where you want to save the repository.

Next, type the following command. Since you have already copied the repository URL, you can paste it after git clone: 

git clone https://github.com/your-username/my-first-repo.git
[What does this command do?]

The git clone command copies the specified repository (in this case, from GitHub) to your local machine. Git automatically sets up everything, so this is the only command needed for cloning.

Finally, navigate to the newly created repository folder and open it in VSCode. The following code helps you to do this process: 

cd my-first-repo
code .
[What does this command do?]
  • The cd [directory] command changes the current directory to the specified location. This exapmle is supposed to set the project name as my-first-repo.
  • The code . command opens VSCode in the current directory.

Tip: You can use Git Bash as a terminal inside VSCode, which is useful for executing Git commands. [How to open it?]

  1. Open VSCode and go to View → Terminal.
  2. If the default terminal is not Bash, click the + button in the terminal panel and select Bash.

3. Freely create/edit files inside your working directory

You can now create or edit files freely. In the example below, we created two text files: test1.txt and test2.txt inside the working directory. Don't forget to save your files before moving to the next step.

4. After making some edits, stage the changes and commit them

Open the Source Control panel in VSCode by clicking the Source Control icon in the left sidebar. Click the Stage All Changes button to add all modified files to the staging area. This button appears when you hover over the Changes section. See the following image for details:

You can also stage files individually by clicking the + icon next to each file name.

Before clicking the Commit button, be sure to enter a commit message describing your changes, such as 'Version 1.1' or a brief summary of the modifications. (If you accidentally click the button without adding a message, you can write one in the newly opened tab and then close it.)

Finally, click blue Commit button.

5. Synchronize your changes with the remote repository

After committing your changes, the blue Commit button will change to a blue Sync button. Click the Sync button to push your changes to the remote repository. To confirm that your changes were successfully updated, check your repository on GitHub.

Second Example: Git Environment with VSCode without remote repository

This example demonstrates how to set up a Git environment without a remote repository. Some developers want to use Git for version control without uploading their code to GitHub. This could be due to security policies that prohibit internet access or simply because the project is too small to require remote management. In such cases, we cannot use GitHub, so we need to set up Git locally. Before starting, it's helpful to know how to open Bash in VSCode. [How to open it?]

  1. Open VSCode and go to View → Terminal.
  2. If the default terminal is not Bash, click the + button in the terminal panel and select Bash.

This example follows these steps:

  1. Create or choose a working directory that is not yet set up with Git.
  2. Run the command git init.
  3. Edit your files and write code as needed.
  4. Stage and commit your changes. Then, repeat from step 3 as needed.

Let's break down the process.

1. Create or Choose a Working Directory

First, decide where you want to store your project. You can create a new folder or use an existing one.

Open a Bash terminal in VSCode, then navigate to the directory you want to use as your working directory. Use the following command to navigate to your chosen directory: 

cd path/to/your/project
[What does this command do?]

The cd [directory] command changes the current directory to the specified location. Please change the path to your working directory.

2. Initialize a Git Repository

To initialize Git in this directory, run:

git init
code .
[What does this command do?]
  • This initializes a hidden .git folder inside your project, which stores all version history and Git settings.
  • The code . command opens VSCode in the current directory with Git enabled.

3. Edit and Modify Files

Now you can create or edit files in your project as needed. In the example below, we created two text files: test1.txt and test2.txt inside the working directory. Don't forget to save your files before moving to the next step.

4. Stage and Commit Changes

Open the Source Control panel in VSCode by clicking the Source Control icon in the left sidebar. Click the Stage All Changes button to add all modified files to the staging area. This button appears when you hover over the Changes section. See the following image for details:

You can also stage files individually by clicking the + icon next to each file name.

Before clicking the Commit button, be sure to enter a commit message describing your changes, such as 'Version 1.1' or a brief summary of the modifications. (If you accidentally click the button without adding a message, you can write one in the newly opened tab and then close it.)

Finally, click blue Commit button.

The blue Commit button changes to blue Publish Branch button, but no need to click. Go back to step 3 and continue coding.

Third Example: How to Link a Local Git Repository to GitHub with VSCode

In the second example, we added Git to an existing local project. In this final example, we will show you how to connect a local Git repository to GitHub. If you use VSCode, the process is quite simple.

This can be achieved by following these steps:

  1. Initialize Git for the project if you haven’t already (follow the steps in Example 2).
  2. Open the project in VSCode (with Git enabled).
  3. Click the blue Publish Branch button and follow the prompts.

Once these steps are completed, your project will be successfully linked to GitHub, and you can manage it as shown in Example 1.

Summary

The examples above cover the basic fundamentals of using Git and GitHub. Since I demonstrated the process using VSCode, most of the commands are simplified and hidden behind the VSCode UI. However, you can accomplish the same tasks using the command line without relying on any UI. This means you can use any text editor, such as Emacs, to manage your Git repository.

Additionally, I haven't covered some important concepts like branching and merging, which are essential features of Git and GitHub.
